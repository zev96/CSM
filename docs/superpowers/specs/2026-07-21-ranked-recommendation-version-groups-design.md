# 排名榜单推荐区：版本组 + 结构化产品卡片（设计方案 v2）

**Date:** 2026-07-21
**Status:** v2.1 — 方向及全部开放问题已由用户确认（2026-07-21），可开工
**审查记录:** 3 个独立视角（工程正确性 / 边界回归 / 需求契合与维护成本），共 CRITICAL×5、MAJOR×12、MINOR×17，关键发现已全部核实并吸收进本版；详见 §13。

## 1. 需求还原

用户要写「十大排名榜单」类文章的产品推荐区。与现有导购文的差异：

1. **单品推荐内容更细分**：每个产品（主推 TOP1、竞品 TOP2…N）是一张结构化卡片——标题行（如 `### 国内外知名品牌-综合性能首选 TOP1. 霍尼韦尔 H-Max`）+ 若干个带加粗小标题的「点」（**市场口碑数据** / **品牌赛道定位** / **分维度硬核测评**……），正文里有行内加粗的关键数据（**703.7 m³/h**、**32.7dB**）。
2. **推荐区有多个「版本」**：版本1·口碑权威型、版本2·功能拆解型……每个版本的点集合、顺序、主推/竞品结构对称性都不同（版本2 主推 7 个点、竞品压缩成 4 个点）。生成时随机抽一个版本。
3. **版本必须统一**：主推抽到版本1，竞品也必须是版本1；同一版本内部，主推和竞品各自的点再独立随机内容。
4. **点内容也要多版本**：同一个点（如「品牌实力」）可以有多份内容随机抽取。

## 2. 现状与差距

| 需求 | 现状 | 差距 |
| --- | --- | --- |
| 点内容多版本随机 | 基本支持：同筛选多笔记随机 + 笔记内 ①②③ 子素材随机（`sampler.py:76-119`） | 一个例外：解析器无条件剥正文行内加粗（`note_parser.py:107`），示例里的关键数据标粗出不来，需引擎修改（§5.6） |
| 主推卡片多点结构 | 部分支持：`hero_brand` 区域吞并后续段落 | 强制输出「推荐理由：」行（`render.py:186-190`）、无自定义标题行、无小节标签；且按块粒度铺点，每版本 8-10 块，多版本维护成本高 |
| 竞品卡片多点结构 | 不支持：`competitor_pool` 一篇笔记=一个竞品，①②③ 抽 1 段当整段理由 | **核心差距** |
| 多版本结构 | 不支持：模板无版本概念 | **核心差距** |
| 主推/竞品版本联动 | 不支持。跨块联动只有 `follow_slot`；`depends_on`（UI「链接」）校验但不消费，是空壳 | **核心差距** |

关键工程事实（决定方案形态，均已对代码核实）：

- 随机模型：每块独立 RNG，`random.Random(f"{seed}-{block.id}")`（`sampler.py:129`），同 seed 全文可复现；批量候选 seed 间隔 1000（`batch_service.py:324`）；「重新随机」按钮 = seed+1（`article.ts:519-523`）。
- 渲染只遍历 `plan.results`，从不回看 `template.blocks`——版本过滤只需在采样阶段做一次，渲染层天然兼容。
- 笔记 `id = 文件名 stem`（`note_parser.py:88`），reroll 同笔记匹配、兄弟排除、反馈权重聚合全按 note_id——**vault 新增文件的命名必须保证 stem 全库唯一**。
- reroll 用**块级** source 重建候选池（`reroll.py:63-65`），不读 pick.meta——卡片模式必须新开分支。
- 结构面板是扁平数组、整数下标寻址（`TemplateBuilder.vue:187-201`）；嵌套容器 = 状态模型重写，不做。
- vault：素材 = frontmatter + ①②③ 子素材；按文件夹路由（module 为路径子序列）+ frontmatter 等值筛选；筛选属性接口动态聚合，新键自动出现；`note_identity` 品牌有别名折叠、**型号无任何归一化**（`identity.py:86-90`），现存写法有 `竞品-` 前缀习惯。
- 仓库已有「一篇笔记内按 H2 小节匹配」的完整机制（`csm_core/test_framework/section_parser.py`），可复用于竞品卡。

## 3. 方案总览

四件事，全部**加性扩展**、零迁移：

1. **模板级「版本组」+ 块级「版本标签」+ 结构 lint**：模板声明版本选项，生成时按种子一次性抽定；块标注所属版本，采样前过滤。主推/竞品同版本 = 同一次抽签，天然成立；配套保存时的**逐版本结构校验**防漏标（§5.2）。
2. **主推卡片**：`hero_brand` 增加 `sections`（小节计划）。非空即卡片模式——渲染成自包含的结构化卡片（自定义标题行 + 加粗小节 + 每小节独立从素材池随机），不再吞并后续段落、不再输出「推荐理由：」。每版本主推从 8-10 个块缩成 **1 个块**。
3. **竞品卡片**：`competitor_pool` 增加 `sections`。卡片模式下素材形态为「**每竞品一篇卡片笔记**：frontmatter 定身份 + 正文 H2 小节 + 小节内 ①②③ 子素材」。采样 = 按小节覆盖度筛名册 → 抽 N 个竞品 → 逐小节抽子素材。一张超集卡（H2 覆盖多个版本的点）可同时服务多个版本。
4. **配套修缮**：行内加粗保留开关、卡片 reroll 分支、批量/重随的版本锁定语义、覆盖度检查工具。

### 为什么这样切（第一性推导）

- 「主推与竞品同版本」的本质是**一次随机决策、多处消费**。做成模板级抽签 + 块级标签，决策显式落在 plan 里（可展示、可覆盖、可复现），而不是让两个块共享 RNG key 恰好抽到同一下标（不可解释、不可覆盖、素材数量一变就错位）。
- 「版本结构不对称」意味着版本决定**块序列本身**——用「每版本各铺一套块，靠标签互斥」表达，绕开嵌套容器。卡片化把每版本的块数压到 2-3 个，这同时是**防漏标**的关键：标签面缩小 70%+。
- 「竞品卡」的本质是**同一小节计划套在 N 个随机抽出的产品上**。计划属于块配置；产品与内容来自 vault。**运营的天然写作单位是「整张卡」**（需求书里贴的就是整卡），所以竞品素材形态选「单文件 + H2 小节」而不是「每点一文件」——9 竞品 × 2 版本从 81 篇小笔记降到 ≤18 张卡，且避开 note-id 撞名（§13-C1）。
- 主推与竞品的素材形态**故意不对称**：主推是固定单品、素材已按点分池（品牌背书/核心技术/…），小节从池里抽还能跨笔记组合；竞品是长尾多产品，整卡书写成本最低。

### 校验新增（保存时，`Template._validate_structure` + 新增结构 lint）

引用完整性：
- 版本组 id 唯一、options 非空且组内唯一；`disabled_options ⊆ options`。
- 块的 `version_group` 指向已声明组（未填且仅一组 → 默认该组；多组且 `versions` 非空未指明 → 报错）；`versions` 值 ∈ options。
- 卡片模式竞品池：`source` 必须 notes_query 且 module 非空、filter 非空（名册筛选，防 legacy 混入）、至少 1 个 required 小节、**强制不重复竞品**（unique 开关锁死）。

**逐版本结构 lint**（对每个 option 模拟过滤后的序列检查，问题以警告/错误呈现——这是「版本统一」的防错层，机制层之外必须有）：
- 未标版本的块被夹在某版本的 hero 卡…竞品池之间 → 警告「疑似漏标」。
- 某 option 没有任何块引用 → 警告「该版本无专属内容」（抽中时推荐区消失）。
- `follow_slot` / `depends_on` 的目标块，必须在引用块可见的每个版本中同样可见 → 否则报错（防「缺数据」占位符静默上文）。
- 卡片竞品池之前没有可见的 hero 卡 → 警告（TOP 编号将从 1 开始）。
- 同一榜单区内混用 legacy hero 与卡片池（或反之） → 报错。

## 5. 引擎变更

### 5.1 版本抽签与过滤（`assemble_plan`）

1. 采样前对每个版本组：`rng = random.Random(f"{seed}::version::{group.id}")`（`::` 分隔符不可能与块 RNG key `f"{seed}-{block.id}"` 撞串）→ 从 `options - disabled_options` 均匀抽一个。支持 `version_overrides: dict[str, str]` 覆盖（对齐 `user_config` 先例：`GenerateBody` → `GenerateRequest` → `assemble_plan`；**`batch_service.py` 的直调点同样要接**）。
2. **批量候选版本一致**：版本抽签使用**批次基准 seed**（`state.seed`），不用候选偏移 seed——否则 K 个候选各落各的版本，评分（扣分制、不按篇幅归一）会系统性偏袒短版本，候选对比失去意义。`assemble_plan` 增加 `version_seed: int | None` 参数，批量传共享基准。
3. `plan.version_choices: dict[str, str]` 记录抽签结果（前端展示、复现、rerun 锁定、候选落盘随带）。
4. 过滤：`active = [b for b in blocks if not b.versions or choices[group_of(b)] in b.versions]`，之后一切照旧。块 RNG 以 block.id 为键，过滤不影响其他块的随机流；无版本组模板走恒真过滤，**同 seed 输出逐字节不变**（回归锁）。
5. 渲染层零改动（只遍历 results）。

### 5.2 主推卡片采样与渲染

- 采样：逐 section 走现有 `_sample_notes_source` 机制（module 继承 + filter + pick_notes + 子素材），picks 打平进 BlockResult，meta 带 `{section_index, section_label}`；块 meta 带 `heading_template/tier/label_layout`（渲染器设计上不回看 Block，所有渲染参数必须进 meta——v1 审查发现的通用坑）。
- 渲染：`{heading_template}` 一行 + 逐小节 `**{label}** ：{内容}`（inline）或 `**{label}**\n{内容}`（line）；label 为空只输出内容段。**不输出 reason_label**，不吞并后续块。
- 多段点范式（以版本1「分维度硬核测评」= 4 个独立维度段为例）：

```json
{ "label": "分维度硬核测评", "filter": {"素材类型": "除醛维度"} },
{ "label": "",              "filter": {"素材类型": "消毒维度"} },
{ "label": "",              "filter": {"素材类型": "过敏原维度"} },
{ "label": "",              "filter": {"素材类型": "体验维度"} }
```

  四个维度顺序固定、各自独立随机；若想「N 选 K 且顺序随机」则用单 section + `pick_notes: K`。两种都表达得了，示例写进用户文档。

### 5.3 竞品卡片采样

`sections` 非空时替换现有竞品池采样：

1. **名册**：`index.query(source.module, source.filter)`（filter 必填，如 `素材类型: 竞品卡`，天然隔离 legacy `竞品推荐理由` 笔记；卡片模式**不**套用角度过滤——与 legacy 路径的差异要在代码注释里明示）。逐笔记要求 frontmatter 必须有 品牌+型号，缺失 → 剔除并告警（**告警带完整相对路径**，不是裸文件名）。
2. **身份归一**：按 `note_identity` 分组前，型号做归一化——剥 `竞品-` 前缀、全半角折叠、大小写折叠、去空白。同竞品多张卡允许（互为候选）。归一化实现放 identity 层新函数，名册与 reroll 共用。
3. **覆盖度预检**：解析每张卡的 H2 小节（复用 `section_parser` 的规整+宽松匹配；H2 提取在 raw_body 上做，先于 chrome 清洗），竞品入名册需至少一张卡覆盖全部 required 小节。不合格竞品剔除，`plan.warnings` 列出「竞品 → 缺哪些小节 → 涉及哪些文件」（这就是补素材工作清单）。
4. **抽竞品**：从名册 `rng.sample` 抽 N 个（**恒不重复**）。`pick_notes` 为固定 int 且名册不足 → **硬失败**，专用异常带逐竞品缺料清单的中文信息（榜单类不允许「十大」静默变「七大」；`random_between` 则钳到可用数并告警）。空名册同样走专用异常，不复用「empty pool」这种指不到根因的报错。
5. **逐小节抽内容**：对选中竞品的卡片笔记，逐 section 取对应 H2 段内的 ①②③ 子素材随机抽 `pick_variants` 个；无 ①②③ 标记则整段作为唯一子素材。picks 打平，meta 带 `{competitor_id, competitor_title, tier, section_index, section_label, card_note_id}`。
6. **层级标签**：取该竞品卡片 frontmatter `tier_key` 值；多卡冲突按路径序取首个并告警。
7. **跨池去重与连续编号**（同版本多池：TOP2-3 深结构池 + TOP4-10 浅结构池）：去重发生在**采样期**——`assemble_plan` 把前面卡片池已抽竞品身份集传给后池，后池名册先做排除再抽。明确语义：前池结果变化会整体重排后池（接受，这仍在「同 seed 可复现」契约内）。渲染期只维护区域计数器，绝不在渲染期丢卡（渲染期无 RNG 无索引，丢卡=榜单缩水+pick 寻址错位）。**多池（深浅双池）为 v1 正式功能**：引擎完整支持，UI 引导进 PR3（§7.4）。

### 5.4 卡片渲染与榜单区语义

- **legacy 路径一行不动**：现有 hero 吞并区域、`start_index=2`、reason_label 继承等行为原样保留（三个出厂模板逐字节回归锁）。卡片是 `compose_draft` 的**新分支**：
- 榜单区 v2 语法：`hero卡 → (literal)* → 竞品卡池+`，区域计数器从 hero=1 起，池间连续；遇到 heading / 其他块类型 / 新 hero 区域终止。无 hero 的卡池从 1 编号（保存时已有 lint 警告）。
- 卡与卡之间用池的 `card_separator` 分隔。

### 5.5 Reroll（卡片分支，`reroll.py` 新路径）

现有 reroll 用块级 source 重建池、`_build_pick` 只回填 brand/model——卡片 pick 直接走会抽出别家竞品的笔记且丢光卡片 meta。新增分支：

- 主推卡 pick：按 `pick.meta.section_index` 找回该小节的 module/filter 重建**小节子池**，兄弟排除只看同小节 picks；meta 全量保留。
- 竞品卡 pick：Layer1 = 同卡同 H2 段内其他 ①②③ 子素材；Layer2 = 该竞品其他覆盖该小节的卡片笔记；无候选返回 NoCandidates。**不跨竞品、不跨小节**。
- 「换一个竞品」不进 v1（牵连整卡级联重建），整篇重生成覆盖。
- 版本抽签不参与 reroll。
- 前端 slot 级「重随」对卡片块改为**按小节**发起（不是把 40 个 pick 串行刷一遍）。

### 5.6 行内加粗保留（`note_parser` / 渲染链）

`_clean_chrome` 目前无条件剥 `**…**`（示例卡的关键数据标粗全丢，运营无绕过手段）。改法：解析层保留加粗标记；**消费侧**按块决定——legacy 各路径在采样产出时执行与旧行为完全一致的剥除（逐字节回归锁），卡片模式（hero/竞品 sections）默认保留。风险集中在「legacy 剥除位置迁移后逐字节一致」，测试必须覆盖含加粗素材的 legacy 用例。

**润色链排版保留约束（用户已拍板，正式范围，进 PR2）**：finalize LLM 润色链现有契约只承诺保留信息点，不承诺保留排版。必须给链补硬约束——**不得修改模板产出的结构：标题行（`###`）、加粗（`**`）、小节顺序与分段排版原样保留**，只在段内润色文字。落法：① finalize prompt 增加排版保留条款；② 后置校验——润色前后对卡片区做结构指纹比对（标题行集合、加粗小节名集合、段落数），不一致则拒收该轮润色结果并回退原文（保结构优先于保润色）；③ 测试用含卡片结构的 draft 过链断言指纹不变。

### 5.7 生成侧语义

- 「重新随机」（seed+1）默认携带 `version_overrides = plan.version_choices`——**锁版本换素材**；另设显式「换版本」入口（清空 override 或指定目标版本）。不接线的话，2 版本下用户点一次重随有 50% 概率整篇换结构。
- 文章视图顶部展示本次版本（`plan.version_choices`）；批量候选记录里随带版本信息。

- **文件名必须含型号**（`竞品卡-欧瑞达X9.md`）：笔记 id = 文件名 stem，reroll/反馈权重全按 id 寻址，撞名会串卡（审查 CRITICAL 项）。覆盖度检查工具同时报 stem 冲突。

- 小节内 ①②③ = 该点的多版本内容（用户需求 4 的全部实现）；正文可用 `**加粗**` 标关键数据（§5.6 后保真输出）。
- 内容公约（进共享盘 CLAUDE.md §6）：竞品「分维度硬核测评 / 成本与维护」结尾必带一个明确短板或数据缺失；横评总结带短板提醒。

### 6.3 覆盖度检查工具（PR4）

模板编辑器竞品池/主推卡面板加「覆盖度检查」：先做一次**增量刷新**（不是全量重扫，增量索引在位；也不是读陈旧缓存——按钮语义必须是刷新后再查），然后输出：

- 竞品名册表：文件 → 归一化后判定身份 → H2 覆盖矩阵 → 是否入册/缺什么（型号写歪当场可见）；
- stem 冲突清单；
- 主推各小节的候选笔记数；
- 未被引用的素材类型值。

## 7. 前端变更

1. **模板头部：版本组编辑器**。v1 管理一个版本组，选项 chip 增删改 + 每选项「启用」开关（对应 `disabled_options`）。**PR1 就要把模板级未知字段透传进 save body**（现 `save()` 手拼字段，UI 一保存就会丢 `version_groups`——不是 PR3 的事）。
2. **块级版本标签**：结构列表行加版本徽标；BlockEditor 加「所属版本」多选下拉（复用链接下拉交互）。
3. **结构面板版本视图**：「查看：全部 / 版本N」过滤器。实现约束：过滤视图是**展示层映射**（filtered idx → real idx），move/delete 必须走映射后的真实下标（现 `v-for` 的 `:key="i"` 也要改稳定键，否则过滤切换触发组件串位）；**在版本视图下新建的块自动带上当前版本标签**（最便宜的防漏标手段）；「复制所选块到版本N」必须重造 block id（否则撞 duplicate-id 校验）。
4. **卡片编辑器**：hero 卡 sections 行编辑器（label + 目录 + 筛选值 + 数量 + 子素材数）；竞品卡 sections 行编辑器（label + H2 名 + 必需开关 + 子素材数）+ heading_template 输入（占位符按钮）+ 覆盖度检查按钮。筛选值下拉受 `/api/vault/attributes` 每键 20 值截断影响（素材类型现已 21 值、本方案再加 ~15），需按 module 收窄或新增全量值端点。**深浅双池 UI 引导（用户确认要做）**：同版本允许多个竞品卡池；池编辑器显示「本池排位从 TOP{k} 开始（接续前池）」的实时提示；提供「添加接续池」快捷操作（新池默认复制上一池的 heading_template/label_layout/card_separator，sections 留空由用户配浅结构）；结构 lint 保证池间顺序与去重语义可见。
5. **生成侧**：版本展示 chip、「重随=锁版本」接线、显式「换版本」入口、生成表单「版本：随机/指定」。

## 8. 兼容性与迁移

- 全部字段带默认值，加性 union 扩展，旧模板 JSON 不动即可加载；`sections=[]` 走 legacy 路径。
- 无版本组模板同 seed 输出逐字节不变；legacy hero+pool 渲染路径代码不动，独立回归锁（含加粗剥除迁移的一致性用例）。
- 新旧竞品素材同目录共存：卡片池 filter 必填（如 `素材类型: 竞品卡`）与 legacy 池的 `竞品推荐理由` 互斥；PR4 顺带扫一遍现存模板里 filter 为空的竞品池并提示（空 filter 的 legacy 池会把新卡片笔记当整篇理由混入）。
- **回滚风险须知**：旧版本 app 加载新模板时 pydantic 默认忽略未知字段——versions 全丢、各版本块串烧成一篇。热更新回滚到 pre-version 版本前需提示（发版说明标注；单用户场景接受此风险，不做引擎级防护）。

## 9. 测试策略

- **Schema/校验**：版本组序列化、悬空引用、多组歧义；逐版本结构 lint 的五类场景（漏标夹心块 / 空版本 / follow_slot 跨版本 / 无 hero 卡池 / legacy-卡片混区）。
- **采样**：抽签确定性与 `::version::` 命名空间不撞块 key；`version_overrides` 与批量共享基准 seed（K 候选同版本）；无版本组逐字节回归；名册身份归一（竞品- 前缀/全半角/多卡折叠）；覆盖度预检剔除与告警文案（带路径）；固定数不足硬失败/random_between 钳制告警/空名册专用异常；跨池排除与前池变化重排后池的既定语义；卡片强制去重。
- **渲染**：heading_template 占位符（`{title}` 不追加关键词）；inline/line 两种 label 布局；多段点（带 label 首段 + 空 label 续段）；区域连续编号（hero=1 → 池2,3…→ 第二池接续）；card_separator；legacy hero+pool 逐字节锁；加粗保留（卡片）与剥除（legacy）双向。
- **Reroll**：主推小节子池重建、竞品卡 Layer1/Layer2、meta 全量保留、不跨竞品/小节、跨池排除。
- **端到端**：双版本模板两个 seed 各落一个版本的结构断言；批量 3 候选同版本断言。

## 11. 已否决的备选方案

- **嵌套版本容器块**：结构面板扁平状态模型需整体重写，标签过滤在扁平列表上等价表达同一语义。
- **竞品「每点一文件」文件夹制**（v1 初稿方案，审查后否决）：① 9 竞品 × 2 版本 ≈ 81 篇笔记 / 324 个 frontmatter 字段，隐藏的书写量爆炸；② `口碑.md` × N 文件夹撞 note-id（id=stem），reroll 串卡、反馈权重合桶——修复要么全库改 id 语义（工程量大）要么靠命名纪律（必被违反）；③ 运营的自然单位是整卡。单文件 H2 制在三个维度都占优，且 H2 匹配机制仓库已有先例。
- **vault 素材带「版本」frontmatter**：版本是模板结构概念；卡片按「覆盖哪些点」自动获得版本资格（超集卡跨版本复用），写死版本字段反而阻止复用。
- **共享 RNG key 让主推/竞品抽同一下标**：不可解释、不可覆盖、两侧素材数量不等即错位。
- **每版本一个独立模板**：版本选择发生在选模板时而非生成时，公共块要维护 N 份。

## 12. 已确认决策（用户 2026-07-21 拍板，开放问题全部关闭）

1. **层级标签**：维持人工填写（主推块内手填、竞品卡 frontmatter 手填），不做排位段自动逻辑（§6.1）。
2. **「——」分隔线**：仅为需求文档的书写记号，不是输出格式；卡间分隔用 `card_separator` 默认空行即可。
3. **版本权重**：不做，恒均匀随机；schema 不留权重字段。
4. **润色链排版保留**：必须加硬约束——不得修改模板结构（标题、加粗、排版），prompt 条款 + 结构指纹校验回退，进 PR2（§5.6）。
5. **深浅双池**：要做，含 UI 引导，进 PR3（§5.3-7、§7.4）。
