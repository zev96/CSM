# Phase 1 — 品牌型号记忆库（解析层 + 最小增强）设计稿

- 日期：2026-06-23
- 关系：[创作台升级总体路线图](2026-06-23-creation-studio-upgrade-roadmap-design.md) 的 **Phase 1** 细化。
- 落地决策（已确认）：**解析层 + 最小元数据增强**，不重组 vault。
- 关键约束（已确认）：① 面向**多品牌多型号** → 品牌×型号矩阵建模；② 事实核对是**硬门禁**（拦截导出 + 放行口）；③ 写入器留 Phase 3，Phase 1 **只读 + 缺口体检**。
- 两项设计决策（PM 评审后定，可推翻）：① 事实核对白名单 = **本次注入源材料里的数字 ∪ 该型号 specs ∪ 认证清单**（判"LLM 是否凭空引入"，而非只比参数表）；② 拆 skill 过渡期**保留一个"人设+去AI味"合并 skill** 供单 skill 期使用。

---

## 0. 前置依赖（Phase 0）

P1 依赖 Phase 0 的几项地基；因尚未单独排期，**P1 先吸收以下最小子集**（其余仍归 Phase 0）：

- **受控词表**：`维度`（动力/过滤/防缠绕/显尘/续航/噪音/尘杯/重量/刷头/维护…）、**自家品牌清单**（判定 主推 vs 竞品，不靠文件夹名猜）。
- **数据对账**：`素材类型` 字段在真实库不一致（见 §2.2），P1 解析层绕开它；冲突清单产出后交 Phase 0 统一。
- **别名表**：`CEWEY↔希喂`、`米家↔小米` 等，解析层与 registry 共用。

---

## 1. 目标 & 非目标

**目标**：把分散在 vault 的型号事实（参数 / 技术话术 / 品牌背书 / 竞品介绍 / 测试）由 app 实时解析成可查询的「型号记忆」，并在生成时**精确注入 + 防编造参数/认证**；不重写、不搬动用户的 Obsidian 库。

**非目标（Phase 1 不做）**：角度/标题驱动选材（Phase 2）、skill 链多-pass 执行（Phase 2，本期只拆分 + 打 `role` + 留合并 skill）、App 内写回 vault（Phase 3 写入器）。

## 2. 数据模型

### 2.1 `BrandModelMemory`（解析层产出，非落盘）

解析层（新 package `csm_core/brand_memory/`）给定 `(品牌, 型号)`，组装：

```
BrandModelMemory:
  品牌, 型号, 品类, 角色            # 角色 = 主推 | 竞品（由「自家品牌清单」判定，§0）
  specs:    dict[字段, SpecValue]   # 型号级·事实 ← 产品参数表（含原值/归一/单位/区间/约数标记）
  certs:    list[str]              # 型号级·认证 ← 产品参数「认证检测」字段（纳入事实核对）
  scripts:  dict[维度, list[变体]]  # 型号级 ← 核心/次要技术话术
  endorsements: list[变体]          # 品牌级 ← 品牌背书（定位/认证/口碑/媒体/市场）
  intro:    list[变体]              # 介绍文案 ← 竞品: 竞品推荐内容/竞品-<型号>.md；主推: 希喂推荐内容根 note
  tests:    dict[测试项, 结果]      # 型号级 ← 测试项目模块/品牌产品测试结果
  guardrails: list[str]            # 写作禁区（来自拆出的人设 skill + 通用禁区）
  coverage: dict                   # 缺口体检：每层有无数据、话术条数、缺测试、空参数字段
```

**粒度规则（多品牌多型号）**：`endorsements` 品牌级（同品牌型号共享）；`specs/certs/scripts/intro/tests` 型号级。解析层：`型号记忆 = 该型号 specs/certs/scripts/intro/tests + 其品牌 endorsements`。

### 2.2 解析层映射规则（型号 → 哪些笔记）

| 层 | 来源 | 维度/匹配 |
|---|---|---|
| `specs`/`certs` | `产品模块/<品类>/产品参数/<型号>-产品参数.md` | frontmatter `型号`（增强后）或文件名；`certs` 取「认证检测」单元格 |
| `scripts` | `产品模块/<品类>/<品牌>推荐内容/{核心,次要}技术/*` | **维度 = 文件夹 + 文件名**（⚠ 不用 `素材类型`，见下）；frontmatter `品牌+型号` |
| `endorsements` | `产品模块/<品类>/<品牌>推荐内容/品牌背书/*` | frontmatter `品牌` |
| `intro` | 竞品 `竞品推荐内容/竞品-<型号>.md`；主推 `<品牌>推荐内容` | 文件名 / `型号` |
| `tests` | `测试项目模块/<品类>/品牌产品测试结果/<型号>*` | 文件名 / `型号` |

> ⚠ **`素材类型` 字段在真实库不可靠**：CLAUDE.md 规定它取 `{品牌背书/产品推荐理由/科普原理解析…}`，但实际 `核心技术/动力系统①` 写的是 `素材类型: 动力系统`、`品牌定位①` 是 `素材类型: 品牌定位`、`绿光显尘` 是 `素材类型: 次要技术`——同一字段三种用法。**解析层维度一律取「文件夹+文件名」**；该不一致写入 Phase 0 对账清单。

复用 `csm_core/vault/note_parser`（frontmatter + ①② 变体）；参数表（`## 性能参数` 等 H2 + 两列表）解析借 `test_framework/section_parser` 分节思路 + 两列表→dict。

### 2.3 事实白名单（动态构建，供 §4 核对）

**不是静态参数表**，而是**每次生成动态构建**：

```
白名单 = 本次注入给 LLM 的所有源材料中出现的数字
        ∪ 该型号 specs 表全部有效数值
        ∪ 该型号 certs
```

- 空值/占位（`未说明 / - / 无 / 暂无数据 / 0`）**不入白名单、不参与核对**（戴森V12 真空度=0 这类是缺数据，不是事实）。
- 判据：成稿中的数字/认证若**不在该集合** = LLM 凭空引入 → 越界。
- 这样既挡幻觉，又不误拦话术里大量已审数字（`效率提升约60%`/`555nm`/`1700L/min`/`22项` 等都在注入源里）。

## 3. 最小元数据增强（一次性脚本 + 校验）

- **回填 `品牌`/`型号`**：用**已知品牌前缀表**从文件名剥离品牌、余下为型号（`CEWEYDS18`→CEWEY/DS18，`友望大橘Ultra`→友望/大橘Ultra，`追觅Z20增强款`→追觅/Z20增强款）；**解析不了的列人工复核清单**，不瞎猜。
  - `产品参数`/`核心·次要技术`/`品牌产品测试结果`：补 `品牌`+`型号`。
  - `品牌背书`：只补 `品牌`（品牌级）。
- **别名归一**：维护别名表（§0），解析层与 registry 统一。
- **修 `brand_registry`**：增强后 `build_brand_registry` 在真实库返回 33 型号（今天因缺 `品牌` 键返回空）。
- **同步 vault 的 CLAUDE.md**：把新增 `品牌`/`型号` 键写进 `D:\家电组共享\DATA\CLAUDE.md §3`，否则用户下次跑 Obsidian 整理流程会把它当未知/废弃字段清掉。
- **安全（改的是「家电组共享」团队盘，从重）**：
  - **先整盘备份**；确认该 vault 是否纳入 git（根目录有 `.gitignore`，需核实），无 git 则回滚只靠备份。
  - **先在副本上跑 + 全量 diff 校验**，团队知会后再对真实库执行。
  - 纯增 frontmatter 字段、不动正文/不改目录；脚本幂等、可回滚。

## 4. 注入 + 事实核对（硬门禁）

### 4.1 注入

生成锁定型号时，注入 `{命中维度 scripts + endorsements + specs + certs + guardrails}`，prompt 硬约束「参数/认证只能用给定材料里的，不得新增或改动任何数字」。

- **token 预算（必做）**：一维度 8-11 变体 ×10 维度全注入会爆上下文。Phase 1 裁剪策略：**只注入命中维度**（模板/槽位涉及的维度），**每维度上限 N 条变体**（默认 3，可配），specs 全量、endorsements 取代表 K 条。遵守路线图「上下文有明确来源和硬上限」。
- **feature flag**：`brand_memory.inject` 开关（关 = 今天的行为），便于 A/B 与回滚。
- 改 `csm_core/llm/prompts.PromptInputs`：加可选 `brand_facts`（向后兼容）。
- 改 `generate_service._run_job`：调 LLM 前插「取型号记忆切片」一步（线性插入）。

### 4.2 事实核对（拦截导出 + 放行口）

成稿在**导出前**过核对（新 `csm_core/factcheck/`，feature flag `brand_memory.factcheck`）：

1. **抽取**：数字 + 单位（AW/Pa/kPa/min/L/kg/dB/W/转/μm/%/元/mm/mAh/h…）+ 认证名。
2. **归一**：`35kPa→35000`、`12万转→120000`，对齐白名单口径。
3. **匹配**：与 §2.3 动态白名单比对，容忍 区间/单位换算/约数标记（约/近/起/≤/＜/最高/最低）。
4. **多型号作用域**：对比文按**槽位分型号**——hero 槽数字对 hero 白名单、competitor 槽对各竞品白名单；跨型号对比句按句内就近归属，归属不明才报越界。
5. **未匹配 → 拦截导出**：列 `{数字/认证, 所在句子, 定位, 建议}`，每条给放行口：**①改文案 ②标为通用表述（非型号参数） ③本次放行**。全部清掉才解锁。
   - 放行是**会话级临时**（不写 vault，避免撞 Phase 3 写入器）；记录到日志，待 Phase 3 可固化进白名单。

> **已知局限**：媒体奖项 / 机构名 / 技术黑话（"CCTV推荐""DS 2.0"）这类**非数字、非认证**的编造，Phase 1 暂不门禁（仅靠人设 skill 的禁区约束）；列为后续。

## 5. Skill 解耦（+ `role` 字段，含过渡）

- 产出三个 skill：①**人设**(`role: persona`，口吻/结构) ②**去AI味**(`role: humanize`，24 模式) ③**合并 skill**「家电科普人设+去AI味」(`role: persona`，供 Phase 1 单 skill 期直接用)。
- **品牌事实从所有 skill 删除**（改由记忆库注入）。
- **迁移**：模板 `default_skill_id` 指向 `家电科普博主` 的，回填为**合并 skill**，避免断链 + 不丢去AI味。
- **过渡**：Phase 2 skill 链上线后，模板改挂 `[persona, humanize]` 链，合并 skill 退役。
- skill 实际住在用户的 `D:\家电组共享\DATA\skills`，拆分在那执行；`skills_service`/`SkillEditView` 暴露 `role`（旧 skill 无 `role` 视为 `persona`）。

## 9. 验收标准

1. `build_brand_registry` 真实库返回 33 型号、别名归一。
2. 解析 `(CEWEY, DS18)` 得完整记忆：specs + certs + 10 维度话术 + 5 类背书 + intro + 测试 + 缺口体检。
3. **防幻觉**：指定型号生成，故意诱导 LLM 写白名单外数字/认证 → **导出被拦 + 列出 + 可放行**。
4. **不误拦（关键回归）**：一篇**忠实使用注入话术**的成稿（含 `约60%`/`555nm`/`1700L/min`/`22项` 等话术数字）→ **不被拦**。
5. 对比文：hero 与竞品数字各按自己型号核对，不互相误判。
6. `家电科普博主.md` 拆分 + 迁移后，CEWEY 文章质量不退化、去AI味仍生效（A/B 对比）。
7. 「品牌型号」tab 可浏览记忆完整度 + 注入预览。
8. **量化基线**：抽 5 型号各生成 3 篇，记录 参数错误数（目标 0）、禁区命中数、单篇人工修改时长，作为后续阶段 before/after 基准。
